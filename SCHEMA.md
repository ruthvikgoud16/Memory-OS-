# Valkey Database Schema & Key Space Design - MemoryOS

This document specifies the database structure, namespaces, serialization protocols, and query patterns for the Valkey datastore. 

All storage operations in MemoryOS must strictly adhere to the patterns defined below.

---

## 1. Key Space Summary Table

MemoryOS uses logical segmentation through key prefixes. The standard delimiter is the colon (`:`).

| Key Pattern | Valkey Type | Module | Description |
| :--- | :--- | :--- | :--- |
| `memoryos:session:{session_id}` | Hash | Session | Session metadata (status, timestamp, settings). |
| `memoryos:session:active` | Set | Session | Set of currently active session IDs. |
| `memoryos:session:all` | Set | Session | Set of all historic session IDs. |
| `memoryos:memory:session:{session_id}` | Hash | Memory | Short-term context memory for a specific session. |
| `memoryos:memory:shared` | Hash | Memory | Long-term global shared memory across all agents. |
| `memoryos:memory:shared:tag:{tag_name}` | Set | Memory | Indices mapping tag names to shared memory keys. |
| `memoryos:timeline:stream:{session_id}` | Stream | Timeline | Chronological event stream for a single session. |
| `memoryos:timeline:global` | Stream | Timeline | Aggregated global timeline stream of all events. |
| `memoryos:pubsub:session:{session_id}` | Pub/Sub | Pub/Sub | Dynamic channel for agent-to-agent message passing. |
| `memoryos:pubsub:history:{session_id}` | Stream | Pub/Sub | Archived communication history for UI replay. |
| `memoryos:metrics:global` | Hash | Metrics | System-wide performance and request counters. |
| `memoryos:metrics:agents` | Hash | Metrics | Execution counts mapped per agent name. |

---

## 2. Structure Layouts & Data Types

### 2.1 Sessions Module

#### Session Metadata Hash
- **Key**: `memoryos:session:{session_id}`
- **Type**: Hash
- **Structure**:
  ```json
  {
    "id": "uuid-string",
    "name": "Creative Content Run",
    "status": "active", // options: "active", "completed", "failed"
    "created_at": "1782064614", // UNIX epoch timestamp
    "updated_at": "1782064614",
    "metadata": "{\"goal\": \"Write a post about Valkey\", \"agents\": [\"ResearchAgent\", \"WriterAgent\", \"ReviewerAgent\"]}"
  }
  ```
- **TTL Strategy**: Persistent. Cleaned up explicitly via `DEL` or expired after 30 days if archival is not needed.

#### Session Sets
- **Active Sessions Key**: `memoryos:session:active`
- **All Sessions Key**: `memoryos:session:all`
- **Type**: Set
- **Members**: Raw string UUIDs (`"d3b07384-d113-495a-a309-80fb359a68bc"`)

---

### 2.2 Memory Module

#### Session Context Hash
- **Key**: `memoryos:memory:session:{session_id}`
- **Type**: Hash
- **Structure**:
  - Fields are user-specified memory names (e.g. `"research_facts"`, `"article_draft"`).
  - Values are serialized JSON documents containing memory values and write metadata.
  - **Example Field-Value**:
    - Field: `"research_facts"`
    - Value:
      ```json
      {
        "content": "Valkey is a high-performance key-value datastore fork of Redis.",
        "author": "ResearchAgent",
        "timestamp": 1782064635
      }
      ```

#### Shared Agent Memory Hash
- **Key**: `memoryos:memory:shared`
- **Type**: Hash
- **Structure**:
  - Fields are global memory keys (e.g., `"agent:capabilities"`, `"system:guidelines"`).
  - Values are serialized JSON payloads.
  - **Example Field-Value**:
    - Field: `"agent:capabilities"`
    - Value:
      ```json
      {
        "ReviewerAgent": ["spelling_check", "tone_validation", "structural_grading"]
      }
      ```

#### Tag-Based Index Sets
- **Key**: `memoryos:memory:shared:tag:{tag_name}`
- **Type**: Set
- **Members**: Shared memory field names (e.g., `"agent:capabilities"`).
- **Purpose**: Provides quick queries to find shared memory blocks tagged with specific categories (e.g., tags like `"rules"`, `"system"`, `"schema"`).

---

### 2.3 Event Timeline Module

#### Session Event Stream
- **Key**: `memoryos:timeline:stream:{session_id}`
- **Type**: Stream
- **Entry Structure**:
  - **Stream ID**: Autogenerated (`*`, e.g., `1782064650000-0`)
  - **Fields**:
    - `event_id`: UUID string.
    - `timestamp`: UNIX timestamp float string.
    - `agent_name`: Name of the triggering agent (e.g., `"WriterAgent"`).
    - `event_type`: Category (`"execution_start"`, `"tool_call"`, `"state_update"`, `"error"`, `"completion"`).
    - `payload`: JSON string representation of event data (e.g. input arguments, status results, output details).

#### Global Event Stream
- **Key**: `memoryos:timeline:global`
- **Type**: Stream
- **Entry Structure**: Same as the Session Stream, with the addition of the `session_id` field in each entry. Used to aggregate logging across the system for live updates in the dashboard.

---

### 2.4 Pub/Sub (Real-time Communication)

#### Inter-Agent Communication Channel
- **Channel Name**: `memoryos:pubsub:session:{session_id}`
- **Type**: Pub/Sub Channel
- **Message Serialization Format**: JSON String
  ```json
  {
    "message_id": "uuid-string",
    "from_agent": "ResearchAgent",
    "to_agent": "WriterAgent",
    "message_type": "task_trigger",
    "content": {
      "task": "draft_article",
      "facts_key": "research_facts"
    },
    "timestamp": 1782064660.0
  }
  ```

#### Pub/Sub Message Replay Stream
- **Key**: `memoryos:pubsub:history:{session_id}`
- **Type**: Stream
- **Purpose**: Stores pub/sub messages sequentially so that the frontend can retrieve the timeline of communication historical runs.
- **Entry Structure**: Stream IDs matching auto-generation, and fields matching the JSON keys of the pub/sub packet.

---

### 2.5 Metrics Module

#### Global Performance Hash
- **Key**: `memoryos:metrics:global`
- **Type**: Hash
- **Structure**:
  ```json
  {
    "total_sessions_created": "412",
    "total_events_logged": "3204",
    "total_messages_routed": "1290",
    "total_memory_writes": "944"
  }
  ```

#### Agent Execution Counts Hash
- **Key**: `memoryos:metrics:agents`
- **Type**: Hash
- **Structure**:
  ```json
  {
    "ResearchAgent:invocations": "150",
    "WriterAgent:invocations": "142",
    "ReviewerAgent:invocations": "120",
    "ResearchAgent:errors": "2",
    "WriterAgent:errors": "0",
    "ReviewerAgent:errors": "1"
  }
  ```

## 3. Command Patterns & Data Manipulation

To ensure optimal performance and lock-free execution, the following Command Patterns are prescribed for each action:

### Session Creation
```redis
# 1. Create the session hash
HSET memoryos:session:uuid-1234 status "active" name "Demo Run" created_at "1782064614" updated_at "1782064614" metadata "{}"

# 2. Add to active set and general set
SADD memoryos:session:active "uuid-1234"
SADD memoryos:session:all "uuid-1234"

# 3. Increment session count metric
HINCRBY memoryos:metrics:global "total_sessions_created" 1
```

### Session Teardown (Delete / Archive)
```redis
# 1. Update status to completed
HSET memoryos:session:uuid-1234 status "completed" updated_at "1782064800"

# 2. Remove from active list (leave in 'all' list for history)
SREM memoryos:session:active "uuid-1234"
```

### Writing Event to Timeline
```redis
# Append to session stream
XADD memoryos:timeline:stream:uuid-1234 * event_id "event-789" timestamp "1782064650.0" agent_name "ResearchAgent" event_type "tool_call" payload "{\"tool\": \"web_search\"}"

# Append to global dashboard stream
XADD memoryos:timeline:global * session_id "uuid-1234" event_id "event-789" timestamp "1782064650.0" agent_name "ResearchAgent" event_type "tool_call" payload "{\"tool\": \"web_search\"}"

# Increment event counter metric
HINCRBY memoryos:metrics:global "total_events_logged" 1
HINCRBY memoryos:metrics:agents "ResearchAgent:invocations" 1
```

### Writing Shared Memory with Tags
```redis
# 1. Store memory
HSET memoryos:memory:shared "agent:capabilities" "{\"ReviewerAgent\":[\"evaluation\"]}"

# 2. Index memory under a tag
SADD memoryos:memory:shared:tag:metadata "agent:capabilities"
```

## 4. Eviction and Serialization Policies

### Serialization
- All nested objects (metadata, event payloads, memory objects) must be serialized to JSON and stored as standard UTF-8 Strings.
- Data structures in Python should validate JSON compliance using Pydantic models before writing to Valkey.

### Memory Optimization
- Set active TTL limits on transient keys where applicable. Since Valkey is the exclusive storage engine, data of **inactive** sessions should be compressed or archived, but not aggressively deleted immediately to maintain dashboard history.
- Run Valkey with `maxmemory-policy noeviction` (or `volatile-lru` for transient caches if necessary). Because Valkey is our Primary Database, auto-eviction of metadata could cause critical data loss. System memory limits must be managed by the application through clean API deletion routines (`DELETE /session/delete/{id}`).
